# Upgrade to 0.3.0-preview.19

Status: preview. This release adds explicit schedule mutation capability and permission handling on top of the real Controller crontab integration from `0.3.0-preview.18`, and connects the firewall open-port workbench to a real Controller host listening-socket backend. It does not change the Controller database schema or Agent protocol.

Direct upgrade sources are Controller SQLite schemas 1 through 8 and earlier StackPilot 0.3 preview releases. Production deployment requires Debian 12 or Ubuntu 24.04 LTS with Node.js 22.x and the distribution-provided `crontab` group. Before changing the running release, verify the current SQLite database, create an online backup with its SHA-256 sidecar, and preserve Controller configuration, TLS material and the master key. Then run:

```bash
npm run release:preflight
npm run release:upgrade -- /protected/backups/pre-0.3.0-preview.19.sqlite3
```

Deploy Controller and Web together and synchronize the Controller systemd unit from `0.3.0-preview.18` or later. Confirm authenticated `GET /api/overview/current-user-crontab` returns `200`, an ISO `scannedAt`, and the configured `writeEnabled` value. Schedule writes accept only a user session with CSRF, `schedules:write`, and a one-time reauthentication proof; API tokens remain read-only. When `STACKPILOT_ENABLE_CRONTAB_WRITE=1`, verify create, update, immediate execution, and delete with a disposable non-privileged task, then confirm its managed crontab block is removed.

Deploy Controller and Web together. Confirm authenticated `GET /api/firewall/open-ports` returns `backend: "ss"`, real listeners and a backend collection timestamp. Open `#firewall-open`, inspect desktop and mobile layouts, then observe at least one complete 10-second polling interval. The page intentionally does not mutate UFW or nftables rules; a listening socket does not prove that upstream firewalls permit traffic.

Schema 8 remains unchanged. To roll back, switch the Controller/Web release symlink to the previous compatible release and restart the Controller. Database restoration is required only if a separate schema-changing release was crossed during the same maintenance window.
