# Upgrade to 0.3.0-preview.4

Status: preview. This release adds schema 7 site-management state, refines deployment, rollback, firewall and systemd workbenches, and adds stable physical-host identity so a Controller and its co-located Agent can appear as one monitored host without changing the Agent control identity. It also restores clean locked dependency installation across Linux, Windows and container release jobs.

Direct upgrade sources: Controller SQLite schemas 1 through 6 and StackPilot 0.3 preview releases. Production deployment requires Node.js 22.x. Before changing the running release, verify the current SQLite database, create an online backup with its SHA-256 sidecar, and preserve Controller configuration, TLS material and the master key. Then run the release preflight and upgrade commands from the new release:

```bash
npm run release:preflight
npm run release:upgrade -- /protected/backups/pre-0.3.0-preview.4.sqlite3
```

Deploy in this order: Controller and Web, the site and database helpers used on each node, then Agent. Controller continues to accept protocol 1.0 and 1.1 Agents. Upgraded Agents derive an opaque domain-separated SHA-256 physical-host identifier from the platform machine identity and negotiate the field with the Controller. The raw machine identifier is never transmitted or persisted. Older Agents remain compatible and continue to appear as independent nodes until upgraded. A new Agent retries a heartbeat without the optional identity field when an older Controller rejects it with HTTP 400; server failures and network errors are not replayed. Agent UUIDs, permissions, heartbeats, audits and task routing remain unchanged.

Controller merges a co-located Agent into its local monitoring row only when physical identity, hostname and primary IP match exactly and the candidate is unique. CPU, memory and disk capacity are counted once, while Agent control-channel failures still degrade the merged host and produce a risk. Conflicting clone identities remain visible as separate hosts. Verify the Overview and Hosts surfaces after at least one upgraded Agent heartbeat.

Controller schema 7 has no automatic down migration. To roll back from an upgraded database, stop the new Controller, verify the pre-upgrade backup checksum, switch the release symlink to the previous version, restore the verified SQLite backup, and start the previous Controller. Rolling back only the Agent is protocol-compatible, but the physical-host merge may disappear until the Agent is upgraded again.

Known limits: physical-host identity depends on a correctly unique Linux machine-id, macOS IOPlatformUUID or Windows MachineGuid. Cloned systems that retain the same platform identity are not merged automatically when hostname or primary IP differs, but operators must still regenerate machine identity during image provisioning. Native production deployment remains Linux-only, and this preview does not claim certified Agent/database-helper support beyond the compatibility matrix.
