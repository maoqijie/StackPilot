# Upgrade to 0.3.0-preview.25

Status: preview. This release connects the failed-schedule workbench to real automatic cron and immediate execution records while retaining the real audit and firewall backends, user-scoped idempotency, reauthentication and native cron preflight through `0.3.0-preview.24`. It does not change Controller schema 8 or the Agent protocol version.

Direct upgrade sources are Controller SQLite schemas 1 through 8 and earlier StackPilot 0.3 preview releases. Production deployment requires Debian 12 or Ubuntu 24.04 LTS with Node.js 22.x and the distribution `cron` package. Before changing the running release, preserve the Controller service user's crontab, verify `/usr/bin/crontab`, the `crontab` group and `/var/spool/cron/crontabs`, create an online SQLite backup with its SHA-256 sidecar, and retain Controller configuration, TLS material and the master key. Then run:

```bash
npm run release:preflight
npm run release:upgrade -- /protected/backups/pre-0.3.0-preview.25.sqlite3
```

Deploy Controller and Web together, synchronize Controller sysusers and the base systemd unit, then restart during a controlled maintenance window. Confirm authenticated `GET /api/overview/current-user-crontab` returns an ISO `scannedAt`, `writeEnabled: true`, and typed `lastExecution` values. Create a disposable task that fails harmlessly, wait for cron to run it automatically, confirm `source: "cron"`, its real exit code and output under `#schedule-failed`, then delete the task and verify both its managed crontab row and execution-record directory are gone.

Schema 8 remains unchanged. Before rolling back to a release without the fixed runner, remove or disable managed schedules created or rewritten by this release, or restore the preserved service-user crontab. Then switch the Controller/Web release symlink and base unit and restart Controller. Database restoration is required only if a separate schema-changing release was crossed during the same maintenance window.
