# StackPilot 公开发布改造计划

## 目标

将当前可运行的前端原型和本机 API，逐步改造成一个边界清晰、默认安全、可测试、可部署、适合公开协作的自托管运维面板。

本计划遵循以下原则：

- 每一步都能独立验收和回退，不进行一次性大重写。
- 安全问题优先于目录美化和功能扩展。
- 控制中心（Controller）与被控节点（Agent）最终采用独立进程和最小权限模型。
- 前后端共享接口契约，但不共享运行时实现。
- 默认配置必须安全；高风险能力必须显式开启。

## 总体步骤

共分为 **8 个步骤**。建议每个步骤单独建立分支和 Pull Request，完成验收后再进入下一步。

## 第 1 步：封闭当前原型的高风险入口

### 目标

在不进行大规模目录重构的前提下，修复当前 Vite 代理、开放 CORS、本机绑定判断和任意命令执行组合形成的安全风险，让开发版本默认只允许本机安全使用。

### 任务

- 将前端开发服务器默认绑定地址改为 `127.0.0.1`，需要局域网访问时必须显式传入参数。
- 后端 CORS 改为明确的来源白名单；默认不返回通配符来源。
- 不再以“后端监听地址是 localhost”作为请求方可信的依据。
- 为所有写操作增加服务端认证校验；令牌从环境变量读取，不写入仓库或前端源码。
- crontab 创建、修改、删除和立即执行默认关闭，必须通过独立环境变量显式开启。
- 对请求体设置大小上限，对错误响应隐藏内部实现细节。
- 明确处理反向代理场景，不盲目信任 `X-Forwarded-For` 等客户端可伪造请求头。
- 为权限拒绝、来源拒绝、能力开关和请求体上限添加自动化测试。
- 增加 `.env.example`，只放安全的示例值和配置说明。
- 更新 README，说明开发模式的端口、绑定方式、令牌配置和危险能力开关。

### 验收标准

- 未配置令牌时，危险写操作不可用，并返回明确错误。
- 无有效凭据的写请求返回 `401` 或 `403`。
- 非白名单 Origin 无法通过浏览器跨域访问 API。
- 通过 Vite `/api` 代理不能绕过认证和危险能力开关。
- 默认启动后，其他局域网设备无法直接访问开发服务器。
- 超大 JSON 请求被拒绝，不会无限占用内存。
- 密钥、令牌和本地 `.env` 不进入 Git。
- `npm run lint`、`npm run build` 和新增安全测试全部通过。

### 本步不做

- 不拆分 `App.tsx`。
- 不迁移 monorepo。
- 不实现完整用户系统或 RBAC。
- 不实现远程 Agent。
- 不增加新的运维功能。

## 第 2 步：补齐公开仓库基础设施

### 目标

建立公开协作所需的法律、贡献、安全披露和持续集成基础。

### 任务

- 确认并添加 `LICENSE`。
- 添加 `SECURITY.md`、`CONTRIBUTING.md`、行为准则和 Pull Request 模板。
- 完善 `.gitignore`，排除 IDE、本地状态、日志、构建产物和密钥文件。
- 建立 CI，执行安装、lint、测试、构建和依赖漏洞检查。
- 添加版本策略、支持范围和安全漏洞报告方式。
- 在 README 中明确项目成熟度，避免将原型描述为生产可用。

### 验收标准

- 新贡献者只依据仓库文档即可完成本地运行和验证。
- CI 在干净环境中稳定通过。
- 仓库具备明确许可证和安全披露渠道。

## 第 3 步：迁移为边界清晰的 monorepo

### 目标

保持单仓库协作体验，同时建立前端、控制中心、Agent 和共享契约的独立边界。

### 建议结构

```text
StackPilot/
├─ apps/
│  ├─ web/
│  ├─ controller/
│  └─ agent/
├─ packages/
│  ├─ contracts/
│  └─ config/
├─ deploy/
├─ docs/
└─ tests/
```

### 任务

- 使用 npm workspaces 管理各应用和共享包。
- 将现有前端迁移到 `apps/web`。
- 将现有 API 迁移到 `apps/controller`，保持现有接口行为。
- 创建 `packages/contracts`，承载请求、响应和错误模型。
- 统一根目录的开发、测试、lint 和构建命令。

### 验收标准

- 迁移前后的用户可见行为一致。
- 每个应用可独立构建和启动。
- 根目录一条命令可以启动开发环境。
- 前端不能直接导入 Controller 的内部实现。

## 第 4 步：拆分前端单体文件

### 目标

将超大 `App.tsx` 和全局样式拆成可维护、可测试的领域模块。

### 任务

- 按页面和领域拆分为 `pages`、`features`、`components`、`hooks`、`api` 和 `styles`。
- 把静态演示数据与真实 API 数据分离。
- 提取统一的 API Client、错误处理和认证注入逻辑。
- 建立路由层，避免用一个组件维护所有页面状态。
- 为关键组件、表单校验和 API 状态添加测试。
- 保持现有视觉和交互，避免在结构重构中顺带改版。

### 验收标准

- 不再存在数千行的页面或样式单文件。
- 页面可独立加载和测试。
- 共享组件与领域组件边界明确。
- 关键用户流程具备自动化测试。

## 第 5 步：模块化 Controller 与接口契约

### 目标

将 HTTP、认证、业务服务、系统适配器和数据访问分离，并让接口契约可验证。

### 任务

- 将 Controller 迁移到 TypeScript。
- 分离路由、控制器、服务、仓储、平台适配器和配置模块。
- 使用结构化 schema 校验所有输入、输出和环境变量。
- 建立统一错误模型、请求 ID、结构化日志和健康检查。
- 将 Linux、macOS、Windows 系统命令收敛到明确的平台适配层。
- 用契约测试保证 Web 与 Controller 的 API 一致。

### 验收标准

- 路由层不直接执行系统命令。
- 所有外部输入均经过 schema 校验。
- API 变更会通过类型检查或契约测试暴露。
- 核心服务可在不执行真实系统命令的情况下测试。

## 第 6 步：建立 Controller 与 Agent 架构

### 目标

让中心面板不直接承担被控服务器的系统操作，通过最小权限 Agent 执行受控任务。

### 任务

- 定义节点注册、身份认证、心跳、能力声明、任务下发、状态回传和日志协议。
- Agent 使用独立系统用户运行，默认无 root 权限。
- 采用双向身份验证和可轮换的节点凭据。
- 使用任务类型和参数白名单，不接受任意 shell 字符串作为通用远程接口。
- 对确需提权的操作设计最小化 sudoers/helper 边界。
- 实现超时、取消、幂等、重试和离线恢复策略。
- 定义 Controller 与 Agent 的版本兼容策略。

### 验收标准

- Controller 无需 SSH root 密钥即可管理节点。
- Agent 只能执行声明过的能力。
- 任意 shell 执行不是默认或通用接口。
- 每个远程操作都有请求者、目标、参数、结果和审计记录。

## 第 7 步：完善身份、权限、持久化与审计

### 目标

从临时开发令牌升级为适合多用户运维的安全控制面。

### 任务

- 实现初始化管理员、登录、退出、会话过期和凭据轮换。
- 使用安全密码哈希、HttpOnly/Secure/SameSite Cookie 和 CSRF 防护。
- 建立用户、角色、权限、节点范围和高风险操作二次确认模型。
- 引入数据库迁移、备份和恢复机制。
- 建立不可静默覆盖的审计日志。
- 对登录、令牌和高风险操作实施速率限制。
- 设计密钥加密存储和脱敏展示策略。

### 验收标准

- 最小权限角色无法执行越权操作。
- 会话、CSRF、暴力尝试和密钥泄漏有明确防护。
- 数据库可升级、备份和恢复。
- 高风险操作可完整追溯。

## 第 8 步：生产部署、端到端验证与发布

### 目标

提供可重复、可升级、可诊断的生产安装方式，并建立正式发布门槛。

### 任务

- 提供 Docker Compose 和原生 systemd 安装方案。
- 明确反向代理、TLS、可信代理、端口和防火墙配置。
- 添加数据卷、备份、升级、回滚和卸载说明。
- 建立端到端测试，覆盖登录、节点注册、任务执行、失败恢复和权限拒绝。
- 添加依赖锁定、镜像扫描、SBOM、制品校验和发布签名。
- 建立版本发布、数据库迁移和 Controller/Agent 兼容检查流程。
- 完成生产部署指南、故障排查和安全加固文档。

### 验收标准

- 新服务器可按文档重复安装。
- 升级失败时有经过验证的回滚路径。
- 发布制品经过 CI、安全扫描和端到端测试。
- 默认部署不直接暴露内部 API 或未加密的 Agent 通道。

## 推荐执行规则

- 每一步开始前记录基线：`npm run lint`、`npm run build` 和当前测试结果。
- 每个 Pull Request 只处理当前步骤，避免混入 UI 改版或新功能。
- 涉及认证、权限、命令执行、提权和密钥管理的改动必须增加负向测试。
- 不以客户端隐藏按钮代替服务端授权。
- 不以监听地址、来源 IP 或自定义请求头作为唯一身份依据。
- 所有危险能力都应默认关闭，并有清晰的启用提示与审计记录。
